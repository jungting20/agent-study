# 6장 프레임워크 개발 실습

4장에서는 ReAct, Plan-and-Solve, Reflection 등 여러 에이전트의 핵심 워크플로를 네이티브 코드로 직접 구현해 보았습니다. 이 과정을 통해 에이전트의 내부 실행 로직을 이해할 수 있었습니다. 이어서 5장에서는 "사용자" 관점으로 전환하여 로우코드(low-code) 플랫폼이 제공하는 편리함과 효율성을 경험했습니다.

이 장의 목표는 업계의 몇 가지 주류 **에이전트 프레임워크(agent frameworks)**를 사용하여 신뢰할 수 있는 에이전트 애플리케이션을 효율적이고 표준화된 방식으로 구축하는 방법을 탐구하는 것입니다. 먼저 시장의 주요 에이전트 프레임워크들을 개괄한 다음, 대표적인 몇 가지 프레임워크에 대한 완전한 실습 사례를 통해 프레임워크 기반 개발 모델을 경험해 보겠습니다.

## 6.1 수동 구현에서 프레임워크 개발로

일회성 스크립트 작성에서 성숙한 프레임워크 사용으로 전환하는 것은 소프트웨어 공학 분야에서 중요한 인지적 도약(mental leap)입니다. 우리가 4장에서 작성한 코드는 주로 교육 및 이해를 위한 것이었습니다. 이들은 특정 작업을 잘 완수할 수 있지만, 복잡한 로직을 가진 다양하고 여러 유형의 에이전트를 구축하려 한다면 곧 병목 현상에 직면하게 될 것입니다.

프레임워크의 본질은 검증된 "규격(specifications)" 세트를 제공하는 것입니다. 이는 모든 에이전트에 공통적인 반복 작업(예: 메인 루프, 상태 관리, 도구 호출, 로깅 등)을 추상화하고 캡슐화하여, 우리가 새로운 에이전트를 구축할 때 일반적인 하부 구현이 아니라 고유한 비즈니스 로직에 집중할 수 있도록 돕습니다.

### 6.1.1 에이전트 프레임워크가 필요한 이유

실습을 시작하기 전에, 왜 프레임워크를 사용해야 하는지 명확히 할 필요가 있습니다. 독립적인 에이전트 스크립트를 직접 작성하는 것과 비교할 때, 프레임워크 사용의 가치는 주로 다음과 같은 측면에서 나타납니다.

1. **코드 재사용성 및 개발 효율성 향상**: 이는 가장 직접적인 가치입니다. 우수한 프레임워크는 에이전트 운영의 핵심 루프(Agent Loop)를 캡슐화하는 일반적인 `Agent` 기본 클래스 또는 실행기를 제공합니다. ReAct든 Plan-and-Solve든 프레임워크가 제공하는 표준 컴포넌트를 기반으로 빠르게 구축할 수 있어 반복적인 작업을 피할 수 있습니다.
2. **핵심 컴포넌트의 디커플링(Decoupling) 및 확장성 달성**: 강력한 에이전트 시스템은 여러 개의 느슨하게 결합된 모듈로 구성되어야 합니다. 프레임워크의 디자인은 우리가 다음과 같이 서로 다른 관심사를 분리하도록 강제합니다.
   - **모델 레이어(Model Layer)**: 대규모 언어 모델과의 상호 작용을 담당하며, 다양한 모델(OpenAI, Anthropic, 로컬 모델 등)로 쉽게 교체할 수 있습니다.
   - **도구 레이어(Tool Layer)**: 표준화된 도구 정의, 등록 및 실행 인터페이스를 제공하여 새로운 도구를 추가해도 다른 코드에 영향을 주지 않습니다.
   - **메모리 레이어(Memory Layer)**: 단기 및 장기 메모리를 처리하며, 필요에 따라 다양한 메모리 전략(예: 슬라이딩 윈도우, 요약 메모리 등)을 전환할 수 있습니다. 이러한 모듈식 설계 덕분에 전체 시스템의 확장성이 크게 향상되어 모든 컴포넌트를 간편하게 교체하거나 업그레이드할 수 있습니다.
3. **복잡한 상태 관리의 표준화**: 우리가 `ReflectionAgent`에서 구현한 `Memory` 클래스는 단지 단순한 시작에 불과합니다. 실제 장시간 실행되는 에이전트 애플리케이션에서 상태 관리는 콘텍스트 윈도우 제한, 이력 정보 유지(persistence), 다자간 대화 상태 추적 등의 문제를 처리해야 하는 거대한 과제입니다. 프레임워크는 강력하고 범용적인 상태 관리 메커니즘을 제공하므로 개발자는 매번 이러한 복잡한 문제를 직접 다루지 않아도 됩니다.
4. **관찰 가능성(Observability) 및 디버깅 프로세스 간소화**: 에이전트의 동작이 복잡해질 때, 에이전트의 의사 결정 과정을 이해하는 것이 매우 중요해집니다. 잘 설계된 프레임워크는 강력한 관찰 기능이 내장되어 있습니다. 예를 들어, 이벤트 콜백 메커니즘(Callbacks)을 도입하여 에이전트 라이프사이클의 핵심 노드(예: `on_llm_start`, `on_tool_end`, `on_agent_finish`)에서 로깅이나 데이터 보고를 자동으로 트리거할 수 있어, 에이전트의 전체 실행 궤적을 쉽게 추적하고 디버깅할 수 있습니다. 이는 코드에 `print` 문을 수동으로 추가하는 것보다 훨씬 효율적이고 체계적입니다.

따라서 수동 구현에서 프레임워크 개발로의 전환은 코드 구성의 변화일 뿐만 아니라 복잡하고 신뢰할 수 있으며 유지보수가 용이한 에이전트 애플리케이션을 구축하기 위한 필수적인 경로입니다.

### 6.1.2 주류 프레임워크 선택 및 비교

에이전트 프레임워크의 생태계는 전례 없는 속도로 발전하고 있습니다. LangChain과 LlamaIndex가 1세대 범용 LLM 애플리케이션 프레임워크의 패러다임을 정의했다면,
차세대 프레임워크는 특정 영역의 심도 있는 과제, 특히 **멀티 에이전트 협동(Multi-Agent Collaboration)** 및 **복잡한 워크플로 제어(Complex Workflow Control)**를 해결하는 데 더 집중하고 있습니다.

이 장의 후속 실습 과정에서는 이러한 첨단 분야에서 매우 대표적인 네 가지 프레임워크인 AutoGen, AgentScope, CAMEL, LangGraph에 초점을 맞출 것입니다.
표 6.1에 나와 있듯이 이들의 설계 철학은 서로 다르며, 복잡한 에이전트 시스템을 구현하기 위한 다양한 기술적 경로를 대표합니다.

- **AutoGen**: AutoGen의 핵심 아이디어는 대화를 통해 협동을 달성하는 것입니다<sup>[1]</sup>. 멀티 에이전트 시스템을 여러 개의 "대화 가능한(conversable)" 에이전트로 구성된 그룹 채팅으로 추상화합니다. 개발자는 서로 다른 역할(예: `Coder`, `ProductManager`, `Tester`)을 정의하고 이들 간의 상호 작용 규칙을 설정할 수 있습니다(예: `Coder`가 코드 작성을 마치면 `Tester`가 자동으로 이어받음). 작업 해결 과정은 최종 목표가 달성될 때까지 이들 에이전트가 자동화된 메시지 전달을 통해 그룹 채팅에서 지속적으로 대화하고 협력하며 반복하는 과정입니다.
- **AgentScope**: AgentScope는 멀티 에이전트 애플리케이션을 위해 특별히 설계된 고기능 개발 플랫폼입니다<sup>[2]</sup>. 핵심 특징은 **사용 편의성(ease of use)**과 **엔지니어링(engineering)**입니다. 개발자가 에이전트를 쉽게 정의하고, 통신 네트워크를 구축하며, 전체 애플리케이션 라이프사이클을 관리할 수 있는 매우 친화적인 프로그래밍 인터페이스를 제공합니다. 내장된 **메시지 전달 메커니즘**과 분산 배포 지원은 복잡하고 대규모인 멀티 에이전트 시스템을 구축하고 운영하는 데 매우 적합합니다.
- **CAMEL**: CAMEL은 **역할 수행(Role-Playing)**이라는 새로운 협동 방식을 제공합니다<sup>[3]</sup>. 핵심 개념은 두 에이전트(예: `AI Researcher`와 `Python Programmer`)에게 각자의 역할와 공동의 작업 목표를 설정하기만 하면, 이들이 "**Inception Prompting**"의 안내에 따라 자율적으로 여러 차례 대화를 나누며 서로 영감을 주고 협력하여 작업을 완수한다는 것입니다. 이를 통해 멀티 에이전트 대화 프로세스 설계의 복잡성을 크게 줄여줍니다.
- **LangGraph**: LangChain 생태계의 확장으로서 LangGraph는 에이전트의 실행 프로세스를 **그래프(Graph)**로 모델링하는 색다른 접근 방식을 취합니다<sup>[4]</sup>. 기존 체인 구조에서는 정보가 한 방향으로만 흐를 수 있었습니다. LangGraph는 각 작업(예: LLM 호출, 도구 실행)을 그래프의 **노드(Node)**로 정의하고 **에지(Edge)**를 사용하여 노드 간의 이동 로직을 정의합니다. 이러한 설계는 본질적으로 **순환(Cycles)**을 지원하므로 반복, 수정, 자아 성찰이 포함된 Reflection과 같은 복잡한 워크플로를 예외 없이 간단하고 직관적으로 구현할 수 있게 해줍니다.

다음 섹션에서는 이 네 가지 프레임워크 각각에 대한 완전한 실습 사례를 통해 프레임워크 구동 개발 모델을 깊이 있게 경험해 보겠습니다. **참고로**, 시연되는 모든 프로젝트 소스 파일은 `code` 폴더에 위치하며 본문에서는 원리 부분만 설명합니다.

## 6.2 첫 번째 프레임워크: AutoGen

앞서 언급했듯이 AutoGen의 설계 철학은 "대화를 통한 협력 유도"에 뿌리를 두고 있습니다.
복잡한 작업 해결 과정을 다양한 역할을 가진 에이전트 간의 일련의 자동화된 대화로 영리하게 매핑합니다.
이 핵심 개념을 바탕으로 AutoGen 프레임워크는 지속적으로 발전하고 있습니다.
우리는 `0.7.4` 버전을 예로 들 것입니다. 이 버전은 지금까지 가장 최신 버전이며,
클래스 상속 설계에서 더 유연한 구성 아키텍처(compositional architecture)로 전환하는 중요한 아키텍처 리팩토링을 나타내기 때문입니다.
이 프레임워크를 깊이 있게 이해하고 적용하기 위해서는 먼저 가장 핵심적인 구성 요소와 기본 대화 상호 작용 메커니즘을 설명해야 합니다.

### 6.2.1 AutoGen의 핵심 메커니즘

`0.7.4` 버전의 릴리스는 AutoGen 개발사에서 중요한 이정표이며, 프레임워크의 기본 설계에 있어 근본적인 혁신을 의미합니다. 이번 업데이트는 단순한 기능 추가가 아니라 프레임워크의 모듈성, 동시성 성능, 개발자 경험 향상을 목표로 전체 아키텍처를 재고한 것입니다.

(1) 프레임워크 구조의 진화

그림 6.1에 나와 있듯이, 새 아키텍처에서 가장 눈에 띄는 변화는 명확한 레이어링과 비동기 우선(asynchronous-first) 설계 철학의 도입입니다.

- **레이어드 디자인(Layered Design):** 프레임워크는 두 개의 핵심 모듈로 나뉩니다.
  - `autogen-core`: 프레임워크의 밑바탕으로서 언어 모델과의 상호 작용 및 메시지 전달과 같은 핵심 기능을 캡슐화합니다. 이는 프레임워크의 안정성과 향후 확장성을 보장합니다.
  - `autogen-agentchat`: `core` 모듈 위에 구축되어 대화형 에이전트 애플리케이션 개발을 위한 고수준 인터페이스를 제공하고, 멀티 에이전트 애플리케이션의 개발 프로세스를 간소화합니다. 이러한 레이어링 전략은 각 컴포넌트의 책임을 명확히 하고 시스템 결합도를 낮춥니다.
- **비동기 우선(Asynchronous First):** 새 아키텍처는 비동기 프로그래밍(`async/await`)으로 완전히 전환되었습니다. 멀티 에이전트 협업 시나리오에서 네트워크 요청(예: LLM API 호출)은 시간이 가장 많이 소요되는 주요 작업입니다. 비동기 모드는 시스템이 한 에이전트의 응답을 기다리는 동안 다른 작업을 처리할 수 있게 하므로 스레드 블로킹을 방지하고 동시 처리 능력과 시스템 자원 활용 효율성을 크게 향상시킵니다.

(2) 핵심 에이전트 컴포넌트

에이전트는 작업을 실행하는 기본 단위입니다. `0.7.4` 버전에서 에이전트 설계는 더 집중적이고 모듈화되었습니다.

- **AssistantAgent (어시스턴트 에이전트):** 주요 작업 해결사로서, 핵심은 대규모 언어 모델(LLM)을 캡슐화하는 것입니다. 이 에이전트의 책임은 대화 이력을 바탕으로 계획 제안, 기사 작성, 코드 생성 등 논리적이고 풍부한 지식을 담은 답변을 생성하는 것입니다. 다양한 시스템 메시지(System Message)를 통해 에이전트에게 서로 다른 "전문가" 역할을 부여할 수 있습니다.
- **UserProxyAgent (사용자 프록시 에이전트):** AutoGen에서 기능적으로 독특한 컴포넌트입니다. 이 에이전트는 이중적인 역할을 수행합니다. 인간 사용자의 "대변인"으로서 작업을 시작하고 의도를 전달하는 역할을 함과 동시에, 코드를 실행하거나 도구를 호출하고 결과를 다른 에이전트에게 피드백하도록 구성할 수 있는 신뢰할 수 있는 "실행기(executor)" 역할도 합니다. 이 설계는 "사고"( `AssistantAgent`가 수행)와 "행동"을 명확히 구분합니다.

(3) GroupChatManager에서 Team으로

작업을 완료하기 위해 여러 에이전트가 협업해야 할 때, 대화 프로세스를 조정하는 메커니즘이 필요합니다. 이전 버전에서는 `GroupChatManager`가 이 책임을 맡았습니다. 새 아키텍처에서는 `RoundRobinGroupChat`과 같은 보다 유연한 `Team` 또는 그룹 채팅 개념이 도입되었습니다.

- **라운드 로빈 그룹 채팅 (RoundRobinGroupChat):** 명확하고 순차적인 대화 조정 메커니즘입니다. 참여하는 에이전트들이 미리 정의된 순서에 따라 차례대로 발언하도록 합니다. 이 모드는 고정된 프로세스가 있는 작업에 매우 적합합니다. 예를 들어, 전형적인 소프트웨어 개발 프로세스에서 제품 관리자가 먼저 요구사항을 제안하면 엔지니어가 코드를 작성하고 마지막으로 코드 검토자가 이를 확인하는 식입니다.
- **워크플로(Workflow):**
  1. 먼저, `RoundRobinGroupChat` 인스턴스를 생성하고 협업에 참여할 모든 에이전트(예: 제품 관리자, 엔지니어 등)를 추가합니다.
  2. 작업이 시작되면, 그룹 채팅은 미리 설정된 순서에 따라 해당하는 에이전트를 차례로 활성화합니다.
  3. 선택된 에이전트는 현재 대화 콘텍스트를 바탕으로 응답합니다.
  4. 그룹 채팅은 새로운 답변을 대화 이력에 추가하고 다음 에이전트를 활성화합니다.
  5. 이 프로세스는 최대 대화 라운드 수에 도달하거나 사전 설정된 종료 조건이 충족될 때까지 계속됩니다.

이처럼 AutoGen은 복잡한 협업 관계를 프로세스가 명확하고 관리하기 쉬운 자동화된 "원탁 회의"로 단순화합니다. 개발자는 각 팀원의 역할과 발언 순서만 정의하면 되며, 나머지 협업 프로세스는 그룹 채팅 메커니즘에 의해 자율적으로 진행됩니다.

다음 섹션에서는 새 아키텍처에서 서로 다른 역할을 가진 에이전트를 정의하고, 이를 `RoundRobinGroupChat`으로 조정되는 그룹 채팅으로 구성하여 가상의 소프트웨어 개발 팀을 구축하고 실제 프로그래밍 작업을 협업하여 완성하는 과정을 직접 경험해 보겠습니다.

### 6.2.2 가상 소프트웨어 개발 팀 구축

AutoGen의 핵심 컴포넌트와 대화 메커니즘을 이해했으므로, 이 섹션에서는 완전한 실습 사례를 통해 이러한 새로운 기능들을 적용하는 방법을 구체적으로 시연할 것입니다. 우리는 서로 다른 전문 기술을 가진 여러 에이전트로 구성된 가상의 소프트웨어 개발 팀을 구축할 것이며, 이들은 협력하여 실제 소프트웨어 개발 작업을 완수할 것입니다.

(1) 비즈니스 목표

우리의 목표는 **비트코인의 현재 가격을 실시간으로 표시**하는 명확한 기능을 가진 웹 애플리케이션을 개발하는 것입니다. 이 작업은 규모는 작지만 요구사항 분석, 기술 선택, 코딩 구현부터 코드 검토 및 최종 테스트에 이르기까지 일반적인 소프트웨어 개발 단계를 모두 다룹니다. 따라서 AutoGen의 자동화된 협업 프로세스를 테스트하기에 이상적인 시나리오입니다.

(2) 에이전트 팀 역할

실제 소프트웨어 개발 프로세스를 시뮬레이션하기 위해 다음과 같이 명확한 책임을 가진 네 가지 에이전트를 설계했습니다.

- **ProductManager (제품 관리자):** 사용자의 모호한 요구사항을 명확하고 실행 가능한 개발 계획으로 변환하는 역할을 담당합니다.
- **Engineer (엔지니어):** 개발 계획을 바탕으로 구체적인 애플리케이션 코드를 작성하는 역할을 담당합니다.
- **CodeReviewer (코드 검토자):** 엔지니어가 제출한 코드를 검토하여 품질, 가독성 및 견고성을 보장하는 역할을 담당합니다.
- **UserProxy (사용자 프록시):** 최종 사용자를 대변하여 초기 작업을 시작하고, 최종 인도된 코드를 실행하고 검증하는 역할을 담당합니다.

이러한 역할 분담은 멀티 에이전트 시스템 설계의 핵심 단계로, 복잡한 작업을 도메인 "전문가"들이 처리하는 여러 하위 작업으로 세분화합니다.

### 6.2.3 핵심 코드 구현

아래에서는 이 자동화된 팀의 핵심 코드를 단계별로 분석해 보겠습니다.

(1) 모델 클라이언트 구성

모든 LLM 기반 에이전트는 언어 모델과 상호 작용하기 위해 모델 클라이언트가 필요합니다. AutoGen `0.7.4`는 OpenAI API 규격과 호환되는 모든 모델 서비스(OpenAI 공식 서비스, Azure OpenAI, Ollama와 같은 로컬 모델 서비스 등 포함)와 편리하게 연동할 수 있는 표준화된 `OpenAIChatCompletionClient`를 제공합니다.

우리는 독립된 함수를 통해 모델 클라이언트를 생성 및 구성하고, API Key와 서비스 주소는 환경 변수를 통해 관리합니다. 이는 코드의 유연성과 보안을 향상시키는 좋은 엔지니어링 관행입니다.

```python
from autogen_ext.models.openai import OpenAIChatCompletionClient

def create_openai_model_client():
    """OpenAI 모델 클라이언트 생성 및 구성"""
    return OpenAIChatCompletionClient(
        model=os.getenv("LLM_MODEL_ID", "gpt-4o"),
        api_key=os.getenv("LLM_API_KEY"),
        base_url=os.getenv("LLM_BASE_URL", "https://api.openai.com/v1")
    )
```

(2) 에이전트 역할 정의

에이전트 정의의 핵심은 고품질 시스템 메시지(System Message)를 작성하는 데 있습니다. 시스템 메시지는 에이전트에게 "행동 지침"과 "전문 지식 베이스"를 설정하는 것과 같으며, 에이전트의 역할, 책임, 워크플로, 심지어 다른 에이전트와 상호 작용하는 방식까지 정확하게 규정합니다. 잘 설계된 시스템 메시지는 멀티 에이전트 시스템이 효율적이고 정확하게 협업할 수 있도록 보장하는 열쇠입니다. 우리 소프트웨어 개발 팀에서는 각 역할마다 정의를 캡슐화하는 독립적인 함수를 만들었습니다.

**제품 관리자 (ProductManager)**

제품 관리자는 전체 프로세스를 시작하는 역할을 맡습니다. 시스템 메시지는 그의 책임을 정의할 뿐만 아니라 출력 구조를 표준화하고 대화를 다음 단계(엔지니어)로 인도하는 명확한 지침을 포함합니다.

```python
def create_product_manager(model_client):
    """제품 관리자 에이전트 생성"""
    system_message = """당신은 소프트웨어 제품의 요구사항 분석 및 프로젝트 계획 수립을 전문으로 하는 숙련된 제품 관리자입니다.

당신의 핵심 책임은 다음과 같습니다:
1. **요구사항 분석**: 사용자의 요구사항을 깊이 이해하고, 핵심 기능과 경계 조건을 식별합니다.
2. **기술 계획**: 요구사항을 기반으로 명확한 기술 구현 경로를 수립합니다.
3. **위험 평가**: 잠재적인 기술적 위험과 사용자 경험 문제를 식별합니다.
4. **조정 및 커뮤니케이션**: 엔지니어 및 기타 팀원들과 효과적으로 소통합니다.

개발 작업을 전달받으면 다음 구조에 따라 분석을 수행해 주세요:
1. 요구사항 이해 및 분석
2. 기능 모듈 분할
3. 기술 선택 제안
4. 구현 우선순위 정렬
5. 인수 기준 정의

답변은 간결하고 명확하게 작성해 주시고, 분석을 완료한 후에는 "Please engineer start implementation"이라고 말해 주세요."""

    return AssistantAgent(
        name="ProductManager",
        model_client=model_client,
        system_message=system_message,
    )
```

**엔지니어 (Engineer)**

엔지니어의 시스템 메시지는 기술 구현에 초점을 맞춥니다. 엔지니어의 기술적 전문성을 나열하고 작업을 전달받은 후의 구체적인 조치 단계를 명시하며, 프로세스를 코드 검토자에게 전달하는 지침도 포함합니다.

```python
def create_engineer(model_client):
    """소프트웨어 엔지니어 에이전트 생성"""
    system_message = """당신은 Python 개발 및 웹 애플리케이션 구축에 능숙한 시니어 소프트웨어 엔지니어입니다.

당신의 기술적 전문 분야는 다음과 같습니다:
1. **Python 프로그래밍**: Python 문법 및 모범 사례에 정통합니다.
2. **웹 개발**: Streamlit, Flask, Django와 같은 프레임워크 전문가입니다.
3. **API 연동**: 서드파티 API 연동에 대한 풍부한 경험이 있습니다.
4. **오류 처리**: 코드의 견고성과 예외 처리에 집중합니다.

개발 작업을 전달받으면 다음을 수행해 주세요:
1. 기술적 요구사항을 주의 깊게 분석합니다.
2. 적절한 기술 솔루션을 선택합니다.
3. 완전한 코드 구현을 작성합니다.
4. 필요한 주석과 설명을 추가합니다.
5. 경계 케이스 및 예외 처리를 고려합니다.

실행 가능한 전체 코드를 제공하고, 완료 후에는 "Please code reviewer check"라고 말해 주세요."""

    return AssistantAgent(
        name="Engineer",
        model_client=model_client,
        system_message=system_message,
    )
```

**코드 검토자 (CodeReviewer)**

코드 검토자의 정의는 코드 품질, 보안 및 표준화에 중점을 둡니다. 시스템 메시지는 검토 초점과 프로세스를 세부적으로 설명하여 코드 인도 전 품질의 최종 관문 역할을 하도록 합니다.

```python
def create_code_reviewer(model_client):
    """코드 검토자 에이전트 생성"""
    system_message = """당신은 코드 품질과 모범 사례에 집중하는 숙련된 코드 검토 전문가입니다.

당신의 검토 초점은 다음과 같습니다:
1. **코드 품질**: 코드 가독성, 유지보수성 및 성능을 확인합니다.
2. **보안**: 잠재적인 보안 취약점과 위험 요소를 식별합니다.
3. **모범 사례**: 코드가 업계 표준 및 모범 사례를 따르는지 확인합니다.
4. **오류 처리**: 예외 처리의 완전성과 합리성을 검증합니다.

검토 프로세스:
1. 코드 로직을 주의 깊게 읽고 이해합니다.
2. 코드 표준 및 모범 사례를 확인합니다.
3. 잠재적인 문제와 개선점을 식별합니다.
4. 구체적인 수정 제안을 제공합니다.
5. 전반적인 코드 품질을 평가합니다.

구체적인 검토 의견을 제공하고, 완료 후에는 "Code review completed, please user proxy test"라고 말해 주세요."""

    return AssistantAgent(
        name="CodeReviewer",
        model_client=model_client,
        system_message=system_message,
    )
```

**사용자 프록시 (UserProxy)**

`UserProxyAgent`는 답변 생성을 위해 LLM에 의존하지 않고 시스템 내에서 사용자의 프록시 역할을 수행하는 특수한 에이전트입니다. 이 에이전트의 `description` 필드는 해당 책임을 명확히 기술합니다. 특히 중요한 점은 작업이 마침내 성공적으로 완료되었을 때 `TERMINATE` 명령을 내려 전체 협업 프로세스를 정상적으로 종료할 책임을 갖는다는 것입니다.

```python
def create_user_proxy():
    """사용자 프록시 에이전트 생성"""
    return UserProxyAgent(
        name="UserProxy",
        description="""사용자 프록시로서 다음 의무를 담당합니다:
1. 사용자를 대신하여 개발 요구사항을 제시합니다.
2. 최종 코드 구현을 실행합니다.
3. 기능이 기대에 부합하는지 검증합니다.
4. 사용자 피드백 및 제안을 제공합니다.

테스트를 완료한 후 TERMINATE라고 응답해 주세요.""",
    )
```

이 네 가지의 독립적인 정의 함수를 통해 완벽한 기능을 갖춘 "가상 팀"을 구축했을 뿐만 아니라, 시스템 메시지를 통한 "프롬프트 엔지니어링"이 효율적인 멀티 에이전트 애플리케이션을 설계하는 핵심적인 부분임을 보여주었습니다.

(3) 팀 협업 프로세스 정의

이 사례에서 소프트웨어 개발 프로세스는 상대적으로 고정되어 있으므로(요구사항 -> 코딩 -> 검토 -> 테스트), `RoundRobinGroupChat`(라운드 로빈 그룹 채팅)이 이상적인 선택입니다. 비즈니스 로직 순서에 따라 네 가지 에이전트를 참가자 리스트에 추가합니다.

```python
from autogen_agentchat.teams import RoundRobinGroupChat
from autogen_agentchat.conditions import TextMentionTermination

# 팀 채팅 및 협업 규칙 정의
team_chat = RoundRobinGroupChat(
    participants=[
        product_manager,
        engineer,
        code_reviewer,
        user_proxy
    ],
    termination_condition=TextMentionTermination("TERMINATE"),
    max_turns=20,
)
```

- **참가자 순서:** `participants` 리스트의 순서가 에이전트들의 발언 순서를 결정합니다.
- **종료 조건:** `termination_condition`은 협업 프로세스가 언제 끝나는지 제어하는 핵심입니다. 여기서는 메시지에 "TERMINATE" 키워드가 포함되면 대화가 끝나도록 설정했습니다. 우리의 설계에서 이 명령은 최종 테스트를 완료한 후 `UserProxy`가 내립니다.
- **최대 라운드 수:** `max_turns`는 대화가 무한 루프에 빠지는 것을 방지하고 불사용 소모를 피하기 위한 안전장치입니다.

(4) 시작 및 실행

AutoGen `0.7.4`는 비동기 아키텍처를 채택하고 있으므로, 전체 협업 프로세스의 시작과 실행은 비동기 함수 내에서 처리되며 최종적으로 `asyncio.run()`을 통해 실행됩니다.

```python
async def run_software_development_team():
    # ... 클라이언트 및 에이전트 초기화 ...

    # 작업 설명 정의
    task = """실시간 비트코인 가격 표시 애플리케이션을 개발해야 합니다. 세부 요구사항은 다음과 같습니다:
            핵심 기능:
            - 실시간 비트코인 현재 가격 표시(USD)
            - 24시간 가격 변동 추세 표시(등락액 및 등락률)
            - 가격 새로고침 기능 제공

            기술적 요구사항:
            - Streamlit 프레임워크를 사용하여 웹 애플리케이션 생성
            - 간단하고 아름다운 인터페이스, 사용자 친화적 디자인
            - 적절한 오류 처리 및 로딩 상태 추가

            요구사항 분석부터 최종 구현까지 팀이 협업하여 이 작업을 완료해 주세요."""

    # 팀 협업을 비동기적으로 실행하고 대화 프로세스를 스트리밍 출력
    result = await Console(team_chat.run_stream(task=task))
    return result

# 메인 프로그램 엔트리
if __name__ == "__main__":
    result = asyncio.run(run_software_development_team())
```

프로그램이 실행되면 `task`가 초기 메시지로 `team_chat`에 전달되고, 첫 번째 참가자인 제품 관리자가 메시지를 받으면서 전체 자동화 협업 프로세스가 시작됩니다.

### 6.2.4 AutoGen의 장단점 분석

모든 기술 프레임워크는 특정한 적용 시나리오와 설계상의 절충안(trade-offs)을 가지고 있습니다. 이 섹션에서는 AutoGen의 핵심 장점과 실제 애플리케이션에서 마주할 수 있는 한계 및 과제를 객과적으로 분석합니다.

(1) 장점

- 사례에서 볼 수 있듯이, 에이전트 팀을 위해 복잡한 상태 머신이나 제어 흐름 로직을 직접 설계할 필요 없이 완전한 소프트웨어 개발 프로세스를 제품 관리자, 엔지니어, 검토자 간의 대화로 자연스럽게 매핑할 수 있습니다. 이 방식은 인간 팀의 협업 모델에 더 가까우며 복잡한 작업을 모델링하는 장벽을 크게 낮춰줍니다. 개발자는 "어떻게 프로세스를 제어할 것인가(process control)"보다는 "누가(역할)"와 "무엇을 할 것인가(책임)"를 정의하는 데 더 많은 에너지를 집중할 수 있습니다.
- 프레임워크는 시스템 메시지(System Message)를 통해 각 에이전트에게 고도로 전문화된 역할을 부여할 수 있게 해줍니다. 본 사례에서 `ProductManager`는 요구사항에 집중하고, `CodeReviewer`는 품질에 집중합니다. 잘 설계된 에이전트는 서로 다른 프로젝트에서 재사용될 수 있으며 유지보수와 확장이 용이합니다.
- 프로세스 지향적 작업의 경우 `RoundRobinGroupChat`과 같은 메커니즘은 명확하고 예측 가능한 협업 프로세스를 제공합니다. 동시에 `UserProxyAgent` 설계는 "루프 속의 인간(Human-in-the-loop)"을 위한 자연스러운 인터페이스를 제공합니다. 사용자 프록시는 작업의 개시자 역할뿐만 아니라 프로세스의 감독자 및 최종 인수자 역할도 수행할 수 있습니다. 이 설계는 자동화된 시스템이 항상 인간의 감독 하에 있도록 보장합니다.

(2) 한계

- `RoundRobinGroupChat`이 순차적인 프로세스를 제공하더라도, LLM 기반 대화는 본질적으로 불확실성을 수반합니다. 에이전트가 예상을 벗어난 답변을 생성하여 대화가 엉뚱한 방향으로 흐르거나 무한 루프에 빠질 수 있습니다.
- 에이전트 팀의 작업 결과가 기대에 미치지 못할 때 디버깅 과정이 매우 까다로울 수 있습니다. 전통적인 프로그램과 달리 명확한 오류 스택 대신 아주 긴 대화 이력을 마주하게 되기 때문입니다. 이를 "대화형 디버깅(conversational debugging)"의 딜레마라고 부릅니다.

(3) 비-OpenAI 모델을 위한 추가 구성

OpenAI 시리즈가 아닌 다른 모델(예: DeepSeek, 통의천문 등)을 사용하려는 경우, 0.7.4 버전에서는 `OpenAIChatCompletionClient` 파라미터에 모델 정보 딕셔너리를 전달해야 합니다. DeepSeek를 예로 들면 다음과 같습니다.

```python
from autogen_ext.models.openai import OpenAIChatCompletionClient

model_client = OpenAIChatCompletionClient(
    model="deepseek-chat",
    api_key=os.getenv("DEEPSEEK_API_KEY"),
    base_url="https://api.deepseek.com/v1",
    model_info={
        "function_calling": True,
        "max_tokens": 4096,
        "context_length": 32768,
        "vision": False,
        "json_output": True,
        "family": "deepseek",
        "structured_output": True,
    }
)
```

이 `model_info` 딕셔너리는 AutoGen이 모델의 성능 한계를 이해하도록 도와 다양한 모델 서비스에 더 잘 어댑팅할 수 있게 해줍니다.

## 6.3 두 번째 프레임워크: AgentScope

AutoGen의 설계 철학이 "대화를 통한 협력 유도"라면, AgentScope는 또 다른 기술적 경로인 **엔지니어링 우선의 멀티 에이전트 플랫폼**을 대표합니다. 알리바바 다모 아카데미(DAMO Academy)에서 개발한 AgentScope는 대규모의 신뢰성이 높은 멀티 에이전트 애플리케이션을 구축하기 위해 특별히 설계되었습니다. 직관적이고 사용하기 쉬운 프로그래밍 인터페이스를 제공할 뿐만 아니라 분산 배포, 장애 복구, 관찰 가능성 등 기업용 기능이 내장되어 있어 오랫동안 안정적으로 실행되어야 하는 프로덕션 환경 애플리케이션 구축에 매우 적합합니다.

### 6.3.1 AgentScope의 설계

AutoGen과 비교할 때, AgentScope의 핵심적인 차이점은 **메시지 구동형 아키텍처 설계**와 **산업 등급의 엔지니어링 모범 사례**에 있습니다. AutoGen이 유연한 "대화 스튜디오"에 가깝다면, AgentScope는 개발부터 테스트, 배포에 이르기까지 전체 라이프사이클을 지원하는 완전한 "에이전트 운영 체제"를 지향합니다. 많은 프레임워크가 채택한 상속 기반 설계와 달리 AgentScope는 **구성 아키텍처(compositional architecture)**와 **메시지 구동 모드**를 선택했습니다. 이 설계는 시스템의 모듈성을 향상시킬 뿐만 아니라 우수한 동시성 성능과 분산 처리 능력의 기초가 됩니다.

(1) 계층형 아키텍처 시스템

그림 6.2에 나와 있듯이 AgentScope는 명확한 계층형 모듈 설계를 채택하여 최하위 기본 컴포넌트부터 최상위 애플리케이션 오케스트레이션에 이르는 완전한 에이전트 개발 생태계를 형성합니다.

이 아키텍처에서 최하층은 전체 프레임워크의 핵심 빌딩 블록을 제공하는 **기본 컴포넌트(Foundational Components)** 레이어입니다. `Message` 컴포넌트는 단일 텍스트 상호 작용부터 복잡한 멀티모달 콘텐츠까지 지원하는 통일된 메시지 형식을 정의합니다. `Memory` 컴포넌트는 단기 및 장기 메모리 관리를 제공하고, `Model API` 레이어는 다양한 대규모 언어 모델 호출을 추상화하며, `Tool` 컴포넌트는 에이전트가 외부 세계와 상호 작용하는 능력을 캡슐화합니다.

기본 컴포넌트 위에는 보다 고수준의 추상화를 제공하는 **에이전트 수준 인프라(Agent-level Infrastructure)** 레이어가 있습니다. 이 레이어에는 브라우저 사용 에이전트, 딥 리서치 에이전트 등 사전 구축된 에이전트들이 포함되어 있을 뿐만 아니라 클래식 ReAct 패러다임을 구현하여 에이전트 훅(hooks), 병렬 도구 호출, 상태 관리 등의 고급 기능을 지원합니다. 특히 주목할 만한 점은 이 레이어가 **비동기 실행 및 실시간 제어**를 기본적으로 지원한다는 것인데, 이는 다른 프레임워크에 비해 AgentScope가 갖는 중요한 장점입니다.

**멀티 에이전트 협력(Multi-Agent Cooperation)** 레이어는 AgentScope의 핵심 혁신이 드러나는 곳입니다. `MsgHub`는 에이전트 간의 메시지 라우팅 및 상태 관리를 담당하는 메시지 센터 역할을 하며, `Pipeline` 시스템은 순차적(sequential) 및 동시적(concurrent) 등 다양한 실행 모드를 지원하는 유연한 워크플로 오케스트레이션 능력을 제공합니다. 이 설계를 통해 개발자는 복잡한 멀티 에이전트 협업 시나리오를 쉽게 구축할 수 있습니다.

최상위의 **배포 및 개발(Deployment & Development)** 레이어는 엔지니어링을 강조하는 AgentScope의 특징을 반영합니다. `AgentScope Runtime`은 프로덕션 등급의 런타임 환경을 제공하고, `AgentScope Studio`는 개발자에게 완전한 시각적 개발 도구 체인을 지원합니다.

(2) 메시지 구동 (Message-Driven)

AgentScope의 핵심 혁신은 **메시지 구동형 아키텍처**입니다. 이 아키텍처에서 에이전트 간의 모든 상호 작용은 전통적인 함수 호출이 아닌 **메시지(Message)**의 송수신으로 추상화됩니다.

```python
from agentscope.message import Msg

# 메시지의 표준 구조
message = Msg(
    name="Alice",           # 송신자 이름
    content="Hello, Bob!",  # 메시지 내용
    role="user",           # 역할 유형
    metadata={             # 메타데이터 정보
        "timestamp": "2024-01-15T10:30:00Z",
        "message_type": "text",
        "priority": "normal"
    }
)
```

메시지를 상호 작용의 기본 단위로 사용하면 다음과 같은 몇 가지 핵심적인 장점이 있습니다.

- **비동기 디커플링(Asynchronous Decoupling)**: 메시지 송신자와 수신자가 시간적으로 디커플링되어 서로를 기다릴 필요가 없으므로 고동시성(high-concurrency) 시나리오를 자연스럽게 지원합니다.
- **위치 투명성(Location Transparency)**: 에이전트는 다른 에이전트가 로컬 프로세스에 있는지 아니면 원격 서버에 있는지 신경 쓸 필요가 없습니다. 메시지 시스템이 라우팅을 자동으로 처리합니다.
- **관찰 가능성(Observability)**: 모든 메시지를 기록, 추적 및 분석할 수 있어 복잡한 시스템의 디버깅과 모니터링이 크게 간소화됩니다.
- **신뢰성(Reliability)**: 메시지를 영구 저장하고 재시도할 수 있습니다. 시스템 장애가 발생하더라도 최종 일관성을 보장하여 시스템의 내결함성(fault tolerance)을 높입니다.

(3) 에이전트 라이프사이클 관리

AgentScope에서 각 에이전트는 명확한 라이프사이클(초기화, 실행, 일시 중지, 소멸 등)을 가지며 통일된 기본 클래스인 `AgentBase`를 기반으로 구현됩니다. 개발자는 대개 핵심 `reply` 메서드만 주의 깊게 구현하면 됩니다.

```python
from agentscope.agents import AgentBase

class CustomAgent(AgentBase):
    def __init__(self, name: str, **kwargs):
        super().__init__(name=name, **kwargs)
        # 에이전트 초기화 로직

    def reply(self, x: Msg) -> Msg:
        # 에이전트의 핵심 응답 로직
        response = self.model(x.content)
        return Msg(name=self.name, content=response, role="assistant")

    def observe(self, x: Msg) -> None:
        # 에이전트의 관찰 로직 (선택 사항)
        self.memory.add(x)
```

이 설계 패턴은 에이전트 내부의 로직을 외부 통신으로부터 격리합니다. 개발자는 에이전트가 어떻게 "사고하고 응답할지" `reply` 메서드 내에 정의하기만 하면 됩니다.

(4) 메시지 전달 메커니즘

AgentScope에는 전체 메시지 구동형 아키텍처의 허브인 **메시지 센터 (MsgHub)**가 내장되어 있습니다. MsgHub는 메시지 라우팅 및 배포뿐만 아니라 영속성 및 분산 통신과 같은 고급 기능을 통합합니다. 다음과 같은 특징을 갖습니다:

- **유연한 메시지 라우팅**: 일대일(point-to-point), 브로드캐스트, 멀티캐스트 등 다양한 통신 모드를 지원하여 유연하고 복잡한 상호 작용 네트워크를 구축할 수 있습니다.
- **메시지 영속성**: 모든 메시지를 데이터베이스(예: SQLite, MongoDB)에 자동으로 저장하여 장시간 실행되는 작업의 상태를 복구할 수 있습니다.
- **네이티브 분산 지원**: AgentScope의 대표적인 기능입니다. 에이전트를 서로 다른 프로세스나 서버에 배포할 수 있으며, `MsgHub`가 RPC(원격 프로시저 호출)를 통해 크로스 노드 통신을 자동으로 처리하여 개발자에게 이를 완전히 투명하게 만듭니다.

하부 아키텍처에서 제공하는 이러한 엔지니어링 기능 덕분에 AgentScope는 고동시성과 높은 신뢰성을 요구하는 복잡한 애플리케이션 시나리오를 처리할 때 기존 대화 구동형 프레임워크보다 훨씬 유리합니다. 물론 이를 위해 개발자는 메시지 구동의 비동기 프로그래밍 패러다임을 이해하고 적응해야 합니다.

다음 섹션에서는 구체적인 실습 사례인 '삼국지 늑대인간' 게임을 통해 AgentScope 프레임워크의 기능, 특히 동시 상호 작용을 처리하는 우수한 장점을 심도 있게 경험해 보겠습니다.

### 6.3.2 삼국지 늑대인간(Werewolf) 게임

AgentScope의 메시지 구동 아키텍처와 멀티 에이전트 협업 능력을 깊이 이해하기 위해, 우리는 중국의 전통 고전 문화 요소를 융합한 "삼국지 늑대인간" 게임을 구축할 것입니다. 이 사례는 복잡한 멀티 에이전트 상호 작용을 처리하는 AgentScope의 장점을 보여줄 뿐만 아니라, **실시간 협업**, **역할 수행**, **전략적 게임**이 필요한 시나리오에서 메시지 구동 아키텍처의 힘을 완전히 발휘하는 방법을 보여줍니다. 전통적인 늑대인간 게임과 달리, 우리의 "삼국지 늑대인간"은 유비, 관우, 제갈량과 같은 고전 삼국지 인물을 게임에 도입합니다. 각 에이전트는 늑대인간 게임의 기본 작업(늑대인간의 야간 습격, 예언자의 조사, 시민의 추리 등)을 완료해야 할 뿐만 아니라, 그에 매칭되는 삼국지 인물의 개성 및 행동 패턴을 구현해야 합니다. 이 설계를 통해 우리는 AgentScope가 **다층적인 역할 모델링(multi-level role modeling)**을 처리하는 성능을 관찰할 수 있습니다.

(1) 아키텍처 설계 및 핵심 컴포넌트

이 사례의 시스템 설계는 계층형 디커플링 원칙을 따르며, 게임 로직을 세 가지 독립된 레이어로 나눕니다. 각 레이어는 AgentScope의 하나 이상의 핵심 컴포넌트에 매핑됩니다.

- **게임 제어 레이어 (Game Control Layer)**: `ThreeKingdomsWerewolfGame` 클래스가 게임의 메인 컨트롤러 역할을 하며, 글로벌 상태(예: 생존 플레이어 리스트, 현재 게임 단계 등)를 유지하고 게임 프로세스(밤 단계 및 낮 단계 전환)를 진행시키며 승패를 판정합니다.
- **에이전트 상호 작용 레이어 (Agent Interaction Layer)**: 완전히 `MsgHub`에 의해 구동됩니다. 밤에 이루어지는 늑대인간 간의 비밀 협상이든, 낮 동안의 공개 토론이든 에이전트 간의 모든 통신은 메시지 센터를 통해 라우팅되고 배포됩니다.
- **역할 모델링 레이어 (Role Modeling Layer)**: 각 플레이어는 `DialogAgent` 기반의 인스턴스입니다. 정교하게 설계된 시스템 프롬프트를 통해 각 에이전트에게 "게임 역할"과 "삼국지 인물의 개성"이라는 이중 신원을 주입합니다.

(2) 메시지 구동형 게임 흐름

이 사례의 핵심 설계는 게임 흐름을 관리하기 위해 **상태 머신** 대신 **메시지 구동** 방식을 사용하는 것입니다. 기존 구현에서는 게임 단계의 전환이 일반적으로 중앙 집중식 상태 머신에 의해 제어되었습니다. AgentScope 패러다임에서 게임 흐름은 잘 정의된 일련의 메시지 상호 작용 패턴으로 자연스럽게 모델링됩니다.

예를 들어, 늑대인간 단계의 구현은 단순한 함수 호출이 아니라 `MsgHub`를 통해 오직 늑대인간 플레이어들만 포함하는 임시 프라이빗 통신 채널을 동적으로 생성하는 방식으로 처리됩니다.

```python
async def werewolf_phase(self, round_num: int):
    """늑대인간 단계 - 메시지 구동 협업 모드 시연"""
    if not self.werewolves:
        return None

    # 메시지 센터를 통해 늑대인간 전용 통신 채널 구축
    async with MsgHub(
        self.werewolves,
        enable_auto_broadcast=True,
        announcement=await self.moderator.announce(
            f"늑대인간 여러분, 오늘 밤 공격 대상을 토론해 주세요. 생존 플레이어: {format_player_list(self.alive_players)}"
        ),
    ) as werewolves_hub:
        # 토론 단계: 늑대인간들이 메시지를 통해 전략을 교환
        for _ in range(MAX_DISCUSSION_ROUND):
            for wolf in self.werewolves:
                await wolf(structured_model=DiscussionModelCN)

        # 투표 단계: 늑대인간들의 습격 결정을 수집하고 집계
        werewolves_hub.set_auto_broadcast(False)
        kill_votes = await fanout_pipeline(
            self.werewolves,
            msg=await self.moderator.announce("습격 대상을 선택해 주세요"),
            structured_model=WerewolfKillModelCN,
            enable_gather=False,
        )
```

이 설계의 장점은 게임 로직이 딱딱한 상태 전환의 연속이 아니라 "특정 맥락에서 어떤 모드의 메시지 교환을 진행할 것인가"로 명확히 표현된다는 점입니다. 낮 토론(전체 브로드캐스트), 예언자 조사(일대일 요청) 등 다른 단계들도 모두 동일한 설계 패러다임을 따릅니다.

(3) 구조화된 출력을 통한 게임 규칙 제약

늑대인간 게임에서 중요한 과제는 에이전트의 행동이 게임 규칙에 부합하도록 보장하는 것입니다. AgentScope의 **구조화된 출력 메커니즘**은 이 문제에 대한 해답을 제공합니다. 우리는 다양한 게임 행동에 대해 엄격한 데이터 모델을 정의합니다.

```python
class DiscussionModelCN(BaseModel):
    """토론 단계의 출력 형식"""
    reach_agreement: bool = Field(
        description="합의에 도달했는지 여부",
        default=False
    )
    confidence_level: int = Field(
        description="현재 추리에 대한 신뢰도 (1-10)",
        ge=1, le=10,
        default=5
    )
    key_evidence: Optional[str] = Field(
        description="귀하의 관점을 지지하는 핵심 증거",
        default=None
    )

class WitchActionModelCN(BaseModel):
    """마녀 행동의 출력 형식"""
    use_antidote: bool = Field(description="해독제 사용 여부")
    use_poison: bool = Field(description="독약 사용 여부")
    target_name: Optional[str] = Field(description="독약 사용 대상 플레이어 이름")
```

이를 통해 에이전트 출력의 **형식 일관성**을 보장할 뿐만 아니라, 더 중요한 것은 **게임 규칙의 자동화된 제약**을 달성하는 것입니다. 예를 들어, 마녀 에이전트는 동일한 대상에게 해독제와 독약을 동시에 사용할 수 없으며 예언자는 하룻밤에 한 명의 플레이어만 조사할 수 있습니다. 이러한 제약 사항들은 데이터 모델의 필드 정의와 검증 로직을 통해 자동으로 실행됩니다.

(4) 역할 모델링의 이중 과제

이 사례에서 기술적으로 가장 흥미로운 과제는 에이전트가 **게임 기능적 역할**(늑대인간, 예언자 등)과 **문화적 개성 역할**(유비, 조조 등)이라는 두 가지 수준의 역할을 동시에 훌륭하게 수행하도록 만드는 것입니다. 우리는 프롬프트 엔지니어링을 통해 이 문제를 해결했습니다.

```python
def get_role_prompt(role: str, character: str) -> str:
    """역할 프롬프트 획득 - 게임 규칙과 인물 개성의 융합"""
    base_prompt = f"""당신은 {character}이며, 이번 삼국지 늑대인간 게임에서 {role} 역할을 수행하고 있습니다.

중요 규칙:
1. 대화와 추리를 통해서만 게임에 참여할 수 있습니다.
2. 외부 도구나 함수를 호출하려고 시도하지 마세요.
3. 요구된 JSON 형식으로만 정확히 응답해 주세요.

역할 특징:
"""

    if role == "Werewolf":
        return base_prompt + f"""
- 당신은 늑대인간 진영에 속해 있으며, 모든 선량한 시민을 제거하는 것이 목표입니다.
- 밤에는 다른 늑대인간들과 습격 대상에 대해 협상할 수 있습니다.
- 낮에는 신분을 숨기고 시민들을 오도해야 합니다.
- {character}의 개성에 맞게 말하고 행동해 주세요.
"""
```

이 설계 덕분에 우리는 재미있는 현상을 관찰할 수 있습니다. 서로 다른 삼국지 인물들이 동일한 게임 역할을 맡았을 때 완전히 다른 전략과 어조를 보인다는 점입니다. 예를 들어, 늑대인간 역할을 맡은 "조조"는 더욱 교활하고 변장에 능한 모습을 보이는 반면, "장비"는 더욱 직설적이고 충동적인 모습을 보입니다.

(5) 동시 처리 및 내결함성(Fault Tolerance) 메커니즘

AgentScope의 비동기 아키텍처는 이 멀티 에이전트 게임에서 중요한 역할을 합니다. 투표 단계처럼 **여러 에이전트의 결정을 동시에 수집**해야 하는 시나리오가 자주 발생합니다:

```python
# 생존한 모든 플레이어의 투표 결정을 병렬로 수집
vote_msgs = await fanout_pipeline(
    self.alive_players,
    await self.moderator.announce("제거할 플레이어에게 투표해 주세요"),
    structured_model=get_vote_model_cn(self.alive_players),
    enable_gather=False,
)
```

`fanout_pipeline`은 동일한 메시지를 모든 에이전트에게 병렬로 보내고 그들의 응답을 비동기적으로 수집할 수 있게 해줍니다. 이는 게임 실행 효율성을 향상시킬 뿐만 아니라, 실제 늑대인간 게임의 "동시 투표" 시나리오를 시뮬레이션합니다. 동시에 우리는 핵심 지점에 내결함성 처리를 추가했습니다:

```python
try:
    response = await wolf(
        "현재 상황을 분석하고 당신의 관점을 밝혀주세요.",
        structured_model=DiscussionModelCN
    )
except Exception as e:
    print(f"⚠️ 토론 중 {wolf.name} 에러 발생: {e}")
    # 게임의 계속적인 진행을 보장하기 위해 기본 응답 생성
    default_response = DiscussionModelCN(
        reach_agreement=False,
        confidence_level=5,
        key_evidence="일시적으로 분석 불가능"
    )
```

이 설계는 특정 에이전트가 예외 상황을 만나더라도 전체 게임 프로세스가 중단 없이 계속 진행될 수 있도록 보장합니다.

(6) 사례 출력 및 요약

### 6.3.3 AgentScope의 장단점 분석

이 "삼국지 늑대인간" 사례를 통해 우리는 AgentScope 프레임워크의 핵심적인 강점을 깊이 경험했습니다. 이 프레임워크는 메시지 구동형 아키텍처를 기반으로 복잡한 게임 프로세스를 일련의 동시 비동기 메시지 전달 이벤트로 우아하게 매핑하며, 이를 통해 기존 상태 머신의 경직성과 복잡성을 회피합니다. 강력한 구조화 출력 기능과 결합하여 게임 규칙을 코드 수준의 제약 조건으로 직접 전환함으로써 시스템의 안정성과 예측 가능성을 대폭 향상시켰습니다. 이러한 설계 패러다임은 성능 측면에서 네이티브 동시성(concurrency)의 우위를 증명할 뿐만 아니라, 예외 처리 설계 덕분에 단일 에이전트에 예외가 발생하더라도 전체 프로세스가 강건하게 실행되도록 보장합니다.

그러나 AgentScope의 엔지니어링 가치는 일정한 복잡성 비용을 동반합니다. 메시지 구동 아키텍처는 강력하지만 개발자에게 비동기 프로그래밍, 분산 통신 등의 개념에 대한 깊은 이해를 요구하므로 기술 장벽이 높은 편입니다. 간단한 멀티 에이전트 대화 시나리오에서는 이 아키텍처가 과도하게 무겁게 느껴져 "오버엔지니어링(over-engineering)"이 될 위험이 있습니다. 또한 상대적으로 신생 프레임워크인 만큼 생태계와 커뮤니티 리소스가 더 확보되어야 합니다. 따라서 AgentScope는 대규모이자 높은 신뢰성을 요구하는 프로덕션 환경의 멀티 에이전트 시스템에 매우 적합하며, 신속한 프로토타입 개발이나 간단한 애플리케이션 시나리오에서는 보다 경량화된 프레임워크를 선택하는 것이 합리적일 수 있습니다.

## 6.4 세 번째 프레임워크: CAMEL

AutoGen이나 AgentScope 같은 포괄적인 프레임워크와 달리, CAMEL의 원래 핵심 목표는 최소한의 인간 개입으로 두 에이전트가 "역할 수행(role-playing)"을 통해 복잡한 작업을 해결하도록 자율적으로 협업하는 방법을 탐구하는 것입니다.

### 6.4.1 CAMEL의 자율적 협업

CAMEL의 자율적 협업의 초석은 **역할 수행(Role-Playing)**과 **인셉션 프롬프팅(Inception Prompting)**이라는 두 가지 핵심 개념입니다.

(1) 역할 수행 (Role-Playing)

CAMEL의 원래 설계에서 작업은 대개 두 에이전트의 협업으로 완료됩니다. 이 두 에이전트는 서로 보완적이며 명확하게 정의된 "역할"을 할당받습니다. 한쪽은 요구사항 제안, 지침 하달, 작업 단계 구상 등을 담당하는 **"AI 사용자(AI User)"** 역할을 맡고, 다른 한쪽은 주어진 지침에 따라 구체적인 작업을 실행하고 솔루션을 제공하는 **"AI 어시스턴트(AI Assistant)"** 역할을 수행합니다.

예를 들어, "주식 거래 전략 분석 도구 개발"이라는 작업이 있다면:

- **AI 사용자** 역할은 "시니어 주식 거래원"이 될 수 있습니다. 그는 시장과 전략은 잘 이해하지만 프로그래밍은 모릅니다.
- **AI 어시스턴트** 역할은 "우수한 Python 프로그래머"입니다. 그는 코딩 실력은 탁월하지만 주식 거래에 대해서는 아는 바가 없습니다.

이 설계를 통해 작업 해결 과정은 자연스럽게 두 "이종 분야 전문가" 간의 대화로 변환됩니다. 거래원이 전문 요구사항을 제안하면 프로그래머가 이를 코드 구현으로 바꾸어, 두 에이전트가 개별적으로는 완수하기 어려운 복잡한 작업을 함께 완성해 나갑니다.

(2) 인셉션 프롬프팅 (Inception Prompting)

단순히 역할을 설정하는 것만으로는 충분하지 않습니다. 지속적인 인간의 감독 없이도 두 AI가 항상 "역할에 몰입"하고 공통의 목표를 향해 효율적으로 나아가도록 어떻게 보장할 수 있을까요? 이것이 바로 CAMEL의 핵심 기술인 인셉션 프롬프팅이 해결하는 지점입니다. "인셉션 프롬프팅"은 대화가 시작되기 전에 두 에이전트에게 주입되는 정교하게 설계된 구조적 초기 지침(System Prompt)입니다. 이 지침은 에이전트에 이식된 일종의 "행동 프로그램"이며, 일반적으로 다음과 같은 핵심 요소들을 포함합니다:

- **자신의 역할 명시**: 예: "당신은 시니어 주식 거래원입니다..."
- **협력자의 역할 안내**: 예: "당신은 뛰어난 Python 프로그래머와 협력하고 있습니다..."
- **공동 목표 정의**: 예: "여러분의 공동 목표는 주식 거래 전략 분석 도구를 개발하는 것입니다."
- **행동 제약 조건 및 통신 프로토콜 설정**: 가장 중요한 부분입니다. 예를 들어, 지침은 AI 사용자에게 "한 번에 하나의 명확하고 구체적인 단계만 제안할 것"을 요구하고, AI 어시스턴트에게는 "이전 단계를 완료하기 전에 더 많은 세부 정보를 요구하지 말 것"을 지시합니다. 또한 대화의 끝에 작업 완료를 나타내기 위해 두 에이전트가 특정 마커(예: `<SOLUTION>`)를 사용하도록 규정합니다.

이러한 제약 조건들은 대화가 주제에서 벗어나거나 비효율적인 루프에 빠지는 것을 방지하고, 그림 6.3과 같이 고도로 구조화된 작업 중심 방식으로 진행되도록 보장합니다.

### 6.4.2 AI 과학 대중화 전자책

CAMEL 프레임워크의 역할 수행 능력을 깊이 이해하기 위해, 우리는 구체적인 협업 사례를 구축할 것입니다. AI 심리학자와 AI 작가가 협업하여 "미루는 습관의 심리학(The Psychology of Procrastination)"이라는 짧은 대중 과학 전자책을 공동 창작하는 프로젝트입니다. 이 사례는 두 에이전트가 각자의 전문 분야를 활용하여 단일 에이전트가 감당하기 어려운 복잡한 창작 작업을 어떻게 성공적으로 완료할 수 있는지 보여주는 CAMEL의 강점을 잘 담고 있습니다.

(1) 작업 설정

**시나리오 설정**: 심리학에 관심이 있는 대중 독자를 위해 미루는 습관의 심리학에 관한 짧은 전자책을 제작합니다. 과학적 엄밀성과 대중적 가독성을 동시에 갖추어야 합니다.

**에이전트 역할**:

- **심리학자 (Psychologist)**: 심리학 분야의 탄탄한 이론적 배경을 가졌으며 인지 행동 과학, 뇌과학 등 관련 분야에 정통합니다. 전문적인 학술적 통찰과 실증 연구 결과를 지원할 수 있습니다.
- **작가 (Writer)**: 뛰어난 문장력과 서사 능력을 가졌으며 복잡한 학술 개념을 생생하고 이해하기 쉬운 텍스트로 전환하는 데 능숙합니다. 독자 경험과 콘텐츠 가독성에 집중합니다.

(2) 협업 작업 정의

먼저 두 AI 전문가의 공동 목표를 명확히 해야 합니다. 우리는 세부적인 문자열 `task_prompt`를 통해 이 작업을 정의합니다.

```python
심리학에 관심이 있는 일반 독자를 위해 "미루는 습관의 심리학"에 관한 짧은 전자책을 제작해 주세요.
요구사항:
1. 내용은 실증 연구에 기반하여 과학적으로 엄밀해야 합니다.
2. 언어는 이해하기 쉬워야 하며, 과도한 전문 용어 사용을 피해야 합니다.
3. 실용적인 개선 제안과 사례 분석을 포함해야 합니다.
4. 분량은 8000~10000자 정도로 조절해 주세요.
5. 서론, 핵심 장, 결론을 포함하는 명확한 구조를 갖추어야 합니다.
"""

print(Fore.YELLOW + f"협업 작업:\n{task_prompt}\n")
```

`task_prompt`는 전체 협업의 "작업 사양서"입니다. 이는 우리가 달성하고자 하는 목표일 뿐만 아니라 백그라운드에서 CAMEL이 "인셉션 프롬프트"를 생성하는 데도 사용되어 두 에이전트 간의 대화가 항상 이 핵심 목표를 중심으로 진행되도록 보장합니다.

(3) 역할 수행 "사회(Society)" 초기화

다음으로 `RolePlaying` 세션 인스턴스를 생성합니다. 이는 우리가 제공한 역할과 작업을 바탕으로 두 에이전트의 협업 "사회"를 신속하게 구축하는 CAMEL의 핵심 작업입니다.

```python
# 역할 수행 세션 초기화
# AI 작가는 "사용자" 역할을 맡아 집필 구조와 요구사항을 제시하는 책임을 집니다.
# AI 심리학자는 "어시스턴트" 역할을 맡아 전문 지식과 콘텐츠를 제공하는 책임을 집니다.
role_play_session = RolePlaying(
    assistant_role_name="Psychologist",
    user_role_name="Writer",
    task_prompt=task_prompt,
    model=model,
    with_task_specify=False, # 이 예시에서는 제공된 task_prompt를 직접 사용합니다.
)

print(Fore.CYAN + f"세부 작업 설명:\n{role_play_session.task_prompt}\n")
```

`RolePlaying`은 복잡한 프롬프트 엔지니어링을 캡슐화하는 CAMEL 제공 고수준 API입니다. 두 역할의 이름과 작업만 넘겨주면 됩니다. CAMEL 설계에서 `user` 역할은 대화의 "드라이버(driver)"이자 "요청자(demander)"이며, `assistant` 역할은 "실행기(executor)"이자 "솔루션 제공자(solution provider)"입니다. 따라서 구조를 기획하는 "작가"는 `user_role_name`에 배정하고, 전문 지식을 제공하는 "심리학자"는 `assistant_role_name`에 배정합니다.

(4) 자동화 대화 시작 및 구동

마지막으로 전체 대화 프로세스를 구동하는 루프를 작성하여 두 AI 전문가가 자동화된 협업을 시작하도록 합니다.

```python
# 협업 대화 시작
chat_turn_limit, n = 30, 0
# init_chat()을 호출하여 AI가 생성한 대화의 첫 시작 메시지를 획득
input_msg = role_play_session.init_chat()

while n < chat_turn_limit:
    n += 1
    # step() 메서드가 대화의 한 라운드를 구동하며, AI 사용자 및 AI 어시스턴트가 각각 한 번씩 발언
    assistant_response, user_response = role_play_session.step(input_msg)

    # 대화의 조기 종료를 방지하기 위해 메시지가 정상 반환되었는지 확인
    if assistant_response.msg is None or user_response.msg is None:
        break

    print_text_animated(Fore.BLUE + f"Writer (AI User):\n\n{user_response.msg.content}\n")
    print_text_animated(Fore.GREEN + f"Psychologist (AI Assistant):\n\n{assistant_response.msg.content}\n")

    # 작업 완료 플래그 확인
    if "<CAMEL_TASK_DONE>" in user_response.msg.content or "<CAMEL_TASK_DONE>" in assistant_response.msg.content:
        print(Fore.MAGENTA + "✅ 전자책 제작 완료!")
        break

    # 어시스턴트의 답변을 다음 라운드 대화의 입력으로 사용
    input_msg = assistant_response.msg

print(Fore.YELLOW + f"총 {n}라운드의 협업 대화 진행 완료")
```

이 `while` loop는 자동화 협업의 핵심입니다. 대화는 작업과 역할을 바탕으로 `init_chat()` 메서드에 의해 수동 오프닝 없이 자동으로 시작됩니다. 루프의 각 단계는 `step()`을 호출하여 한 라운드의 상호 작용을 진행하며(작가가 요구사항을 제안하고 심리학자가 내용을 제공), 이전 라운드에서 심리학자가 출력한 결과를 다음 라운드의 입력으로 사용하는 창작 체인을 형성합니다. 이 프로세스는 미리 설정된 대화 턴 제한에 도달할 때까지 계속되거나, 한쪽 에이전트가 작업 완료 플래그인 `<CAMEL_TASK_DONE>`을 출력하면 자동으로 종료됩니다.

(5) 협업 프로세스 시연

위 코드를 실행하면 단순한 질의응답 텍스트의 나열을 얻는 데 그치지 않고, 마치 인간 전문가 팀처럼 고도로 구조화된 협업 프로세스가 자동으로 진행되는 것을 확인할 수 있습니다. 전체 창작 과정은 자연스럽게 몇 가지 단계로 나뉩니다:

**1단계 (약 1~5라운드): 프레임워크 구축 및 목표 정렬**
대화 초기에는 "작가" 에이전트가 주도적인 역할을 하여 전자책의 전반적인 구조와 장 배열에 대한 구상을 제시합니다. 이어서 "심리학자"가 전문 지식의 관점에서 이 프레임워크를 검토하고 보완하여 핵심 학술 모듈(이론적 기반, 핵심 개념 등)이 누락되지 않도록 보장함으로써, 협업 초기에 최종 산출물에 대한 합의에 도달합니다.

**2단계 (약 6~20라운드): 핵심 콘텐츠 생성 및 지식 번역**
이 단계는 가장 효율적인 콘텐츠 생성 기간입니다. 협업 모드는 안정적인 "요청-응답" 루프를 이룹니다:
- **심리학자**: "시간 할인 이론(temporal discounting theory)", "실행 기능 결핍(executive function deficits)" 등 핵심 개념에 대한 과학적 설명과 관련 실험 연구 인용을 포함하여 하드코어 전문 지식을 제공하는 역할을 담당합니다.
- **작가**: "번역가" 역할을 맡아 엄격하지만 자칫 모호할 수 있는 학술 개념을 생생하고 비유적인 메타포와 일상적인 사례로 변환합니다. 예컨대 뇌의 "현재 편향(present bias)" 개념을 "장기적인 건강 대신 눈앞의 사탕만 떼쓰며 조르는 철부지 아이"에 비유하는 식입니다.

**3단계 (약 21~25라운드): 반복 최적화 및 품질 보증**
책의 주요 내용이 완성되면 대화의 초점은 기존 텍스트를 다듬고 개선하는 방향으로 전환됩니다. 이 시점에서 두 에이전트의 역할은 미묘하게 바뀝니다:
- **작가**: 글의 전체적인 매끄러움, 논리적 연결성 및 문체를 확인하며 "독자 경험"의 관점에서 수정 사항을 제안하는 데 집중합니다.
- **심리학자**: 다시 "팩트 체커" 역할을 수행하며 텍스트를 다듬는 과정에서 핵심 지식의 과학적 정확성이 손실되지 않았는지 점검하고, 필요한 경우 더 강력한 실증 연구 데이터로 관점을 보완합니다.

**4단계 (종료): 요약 및 고도화**
마지막 몇 라운드의 대화에서 양측은 실용적인 제안들의 요약과 책 전체의 검토를 공동으로 완료하여, 전자책이 독자에게 깊은 인상을 남기고 실질적인 가치를 제공하는 명확하고 강력한 결말로 이어지도록 보장합니다.

```
협업 작업:
심리학에 관심이 있는 일반 독자를 위해 "미루는 습관의 심리학"에 관한 짧은 전자책을 제작해 주세요.
...

세부 작업 설명:
일반 독자를 위한 8000~10000자 분량의 짧은 전자책 "미루는 습관의 심리학" 집필: 실증 연구 기반, 이해하기 쉬운 스타일. 구조: 서론, 원인(인지/감정/보상), 동기 및 의사 결정, 습관 형성 및 개입, 실용 전략 및 실습, 세 가지 사례 분석, 요약 및 참고 자료. 각 장은 연구 인용과 실행 가능한 단계를 포함할 것.
```

### 6.4.3 CAMEL의 장단점 분석

전자책 창작 사례를 통해 CAMEL 프레임워크의 독특한 역할 수행 패러다임을 경험해 보았습니다. 이제 이 설계 철학의 장점과 한계를 객관적으로 분석하여 실제 프로젝트에서 현명한 기술적 선택을 할 수 있도록 돕겠습니다.

(1) 장점

CAMEL의 가장 큰 장점은 "가벼운 아키텍처, 묵직한 프롬프팅(light architecture, heavy prompting)" 설계 철학에 있습니다. AutoGen의 복잡한 대화 관리나 AgentScope의 분산 아키텍처와 비교하여, CAMEL은 정교하게 설계된 초기 프롬프트만으로 높은 수준의 에이전트 협업을 구현할 수 있습니다. 이 자율적으로 발현되는 협업 행동은 하드코딩된 워크플로보다 더 유연하고 효율적입니다.

또한 CAMEL 프레임워크는 빠르게 발전하고 있습니다. 공식 [GitHub 리포지토리](https://github.com/camel-ai/camel)를 보면 CAMEL이 단순한 2인 에이전트 협업 도구를 넘어 다음과 같은 기능들을 탑재하고 있음을 확인할 수 있습니다:

- **멀티모달 기능**: 텍스트, 이미지, 오디오 등 다중 모달리티에서의 에이전트 협업 지원
- **도구 통합**: 검색, 계산, 코드 실행 등을 포함한 풍부한 도구 라이브러리 내장
- **모델 어댑테이션**: OpenAI, Anthropic, Google 및 오픈소스 모델 등 다양한 LLM 백엔드 지원
- **생태계 연계**: LangChain, CrewAI, AutoGen 등 주요 프레임워크와의 상호 운용성 확보

(2) 주요 한계

1. 프롬프트 엔지니어링에 대한 높은 의존도

CAMEL의 성공은 초기 프롬프트의 품질에 크게 의존합니다. 이는 다음과 같은 과제를 낳습니다:

- **프롬프트 설계 장벽**: 대상 도메인과 LLM의 행동 특징에 대한 깊은 이해가 필요합니다.
- **디버깅의 복잡성**: 협업이 비효율적일 때 문제의 원인이 역할 정의인지, 작업 설명인지, 혹은 상호 작용 규칙 때문인지 짚어내기 어렵습니다.
- **일관성 과제**: 서로 다른 LLM이 동일한 프롬프트를 다르게 해석할 수 있습니다.

2. 협업 규모의 제약

CAMEL은 2인 에이전트 협업에서 탁월하지만, 대규모 멀티 에이전트 시나리오를 다룰 때는 과제에 직면합니다:

- **대화 관리**: AutoGen과 같은 복잡한 대화 라우팅 메커니즘이 부족합니다.
- **상태 동기화**: AgentScope와 같은 분산 상태 관리 기능이 없습니다.
- **갈등 해결**: 여러 에이전트의 의견이 엇갈릴 때 효과적인 중재 메커니즘이 부족합니다.

3. 작업 적합성의 경계

CAMEL은 깊은 협업과 창의적 사고가 필요한 작업에 특히 적합하지만, 다음과 같은 시나리오에서는 최선의 선택이 아닐 수 있습니다:

- **엄격한 프로세스 제어**: 정확한 단계별 제어가 필요한 작업에는 LangGraph의 그래프 구조가 더 적합합니다.
- **대규모 동시성**: 고동시성 시나리오에서는 AgentScope의 메시지 구동 아키텍처가 더 유리합니다.
- **복잡한 의사 결정 트리**: 다자간 의사 결정 시나리오에서는 AutoGen의 그룹 채팅 모드가 더 유연합니다.

전반적으로 CAMEL representa는 독특하고 우아한 멀티 에이전트 협업 패러다임을 제안합니다. 인간 중심의 역할 수행 설계를 통해 복잡한 시스템 엔지니어링 문제를 직관적인 대인 협업 모델로 성공적으로 전환합니다. 생태계가 점차 정비되고 기능이 확장됨에 따라, CAMEL은 지능형 협업 시스템을 구축하기 위한 중요한 대안으로 자리 잡고 있습니다.

## 6.5 네 번째 프레임워크: LangGraph

### 6.5.1 LangGraph 구조 개요

LangGraph는 LangChain 생태계의 핵심적인 확장으로 에이전트 프레임워크 설계에 완전히 새로운 방향을 제시합니다.
앞서 소개한 "대화" 기반 프레임워크(AutoGen, CAMEL 등)와 달리,
LangGraph는 에이전트의 실행 흐름을 **상태 머신(State Machine)**으로 모델링하고 이를 **방향성 그래프(Directed Graph)**로 표현합니다.
이 패러다임에서 그래프의 **노드(Node)**는 특정 연산 단계(LLM 호출, 도구 실행 등)를 나타내고, **에지(Edge)**는 한 노드에서 다른 노드로 넘어가는 전이 로직을 정의합니다.
이 설계의 혁신적인 측면은 그래프 구조가 본질적으로 순환(loop)을 지원하므로, 에이전트가 반복적으로 성찰하고 스스로 교정할 수 있는 복잡한 워크플로를 직관적이고 직설적으로 구축할 수 있게 한다는 점입니다.

LangGraph를 이해하려면 먼저 세 가지 기본 컴포넌트를 파악해야 합니다.

**첫째는 전역 상태(State)입니다.** 전체 그래프의 실행 프로세스는 공유되는 하나의 상태 객체를 중심으로 작동합니다.
이 상태는 일반적으로 Python의 `TypedDict`로 정의되며 대화 이력, 중간 결과, 반복 횟수 등 추적해야 하는 모든 정보를 담을 수 있습니다. 모든 노드는 이 중앙 상태를 읽고 업데이트할 수 있습니다.

```python
from typing import TypedDict, List

# 전역 상태 데이터 구조 정의
class AgentState(TypedDict):
    messages: List[str]      # 대화 이력
    current_task: str        # 현재 작업
    final_answer: str        # 최종 답변
    # ... 추적할 기타 상태들
```

**둘째는 노드(Nodes)입니다.** 각 노드는 현재 상태를 입력받아 업데이트된 상태를 출력하는 Python 함수입니다. 노드는 구체적인 작업을 실행하는 단위입니다.

```python
# "기획자(planner)" 노드 함수 정의
def planner_node(state: AgentState) -> AgentState:
    """현재 작업을 바탕으로 계획을 수립하고 상태를 업데이트합니다."""
    current_task = state["current_task"]
    # ... LLM을 호출하여 계획 생성 ...
    plan = f"작업 '{current_task}'에 대해 계획이 생성되었습니다..."

    # 새로운 메시지를 상태에 추가
    state["messages"].append(plan)
    return state

# "실행기(executor)" 노드 함수 정의
def executor_node(state: AgentState) -> AgentState:
    """최신 계획을 실행하고 상태를 업데이트합니다."""
    latest_plan = state["messages"][-1]
    # ... 계획을 실행하고 결과 획득 ...
    result = f"계획 '{latest_plan}' 실행 결과..."

    state["messages"].append(result)
    return state
```

**셋째는 에지(Edges)입니다.** 에지는 노드들을 연결하고 워크플로의 방향을 정의하는 역할을 합니다. 가장 간단한 에지는 일반 에지(regular edge)로, 한 노드의 출력이 항상 고정된 다음 노드로 흐르도록 규정합니다. LangGraph의 가장 강력한 기능은 **조건부 에지(Conditional Edges)**에 있습니다. 이는 상태를 판단하는 함수를 사용하여 현재 전역 상태를 검사하고, 그 결과에 따라 동적으로 어떤 노드로 점프할지 결정합니다. 이것이 순환 및 복잡한 조건 분기 로직을 구현하는 열쇠입니다.

```python
def should_continue(state: AgentState) -> str:
    """조건 함수: 상태에 따라 다음 라우트 결정"""
    # 대화 메시지가 3개 미만이면 계속 기획을 하도록 결정한다고 가정
    if len(state["messages"]) < 3:
        # 반환되는 문자열은 조건부 에지를 추가할 때 정의한 키와 매칭되어야 함
        return "continue_to_planner"
    else:
        state["final_answer"] = state["messages"][-1]
        return "end_workflow"
```

상태, 노드, 에지를 정의하고 나면, 블록을 쌓듯 이들을 조립하여 실행 가능한 워크플로를 구성할 수 있습니다.

```python
from langgraph.graph import StateGraph, END

# 상태 그래프 초기화 및 상태 구조 바인딩
workflow = StateGraph(AgentState)

# 노드 함수들을 그래프에 추가
workflow.add_node("planner", planner_node)
workflow.add_node("executor", executor_node)

# 그래프 엔트리 포인트 설정
workflow.set_entry_point("planner")

# 일반 에지 추가 - 기획자와 실행기 연결
workflow.add_edge("planner", "executor")

# 조건부 에지 추가 - 동적 라우팅 구현
workflow.add_conditional_edges(
    # 시작 노드
    "executor",
    # 판단 함수
    should_continue,
    # 라우트 매핑: 판단 함수의 반환값을 대상 노드에 매핑
    {
        "continue_to_planner": "planner", # "continue_to_planner"가 반환되면 planner 노드로 점프
        "end_workflow": END               # "end_workflow"가 반환되면 프로세스 종료
    }
)

# 그래프 컴파일 - 실행 가능한 애플리케이션 생성
app = workflow.compile()

# 그래프 실행
inputs = {"current_task": "최근 AI 업계 동향 분석", "messages": []}
for event in app.stream(inputs):
    print(event)
```

### 6.5.2 3단계 Q&A 어시스턴트

LangGraph의 핵심 개념을 이해했으므로, 실습 사례를 통해 배운 내용을 다져보겠습니다. 우리는 명확하고 고정된 3단계 프로세스를 따라 사용자의 질문에 답하는 단순화된 질의응답 대화 어시스턴트를 구축할 것입니다:

1. **이해 (Understand)**: 사용자의 의도를 분석합니다.
2. **검색 (Search)**: 의도와 관련된 정보를 인터넷에서 검색하는 과정을 시뮬레이션합니다.
3. **답변 (Answer)**: 분석된 의도와 검색된 정보를 바탕으로 최종 답변을 생성합니다.

이 사례는 상태를 정의하고, 노드를 생성하고, 이를 선형 그래프로 연결하는 워크플로를 잘 보여줄 것입니다. 코드는 전역 상태 정의, 노드 생성, 그래프 구축, 애플리케이션 실행의 4가지 핵심 단계로 나뉩니다.

(1) 전역 상태 정의

먼저 전체 워크플로를 관통하는 전역 상태를 정의해야 합니다. **이는 그래프의 각 노드 사이에서 전달되며 워크플로의 영구적인 콘텍스트 역할을 하는 공유 데이터 구조입니다.** 각 노드는 이 구조에서 데이터를 읽거나 업데이트할 수 있습니다.

```python
from typing import TypedDict, Annotated
from langgraph.graph.message import add_messages

class SearchState(TypedDict):
    messages: Annotated[list, add_messages]
    user_query: str      # LLM이 파악한 사용자 요구사항 요약
    search_query: str    # Tavily API용으로 최적화된 검색어
    search_results: str  # Tavily 검색이 반환한 결과
    final_answer: str    # 최종 생성된 답변
    step: str            # 현재 단계를 표시하는 마크
```

우리는 `SearchState` `TypedDict`를 만들어 상태 객체의 명확한 데이터 스키마를 정의했습니다. 핵심 설계는 `user_query`와 `search_query` 필드를 분리하여 포함한 것입니다. 이를 통해 에이전트는 사용자의 자연어 질문을 검색 엔진에 더 적합한 정제된 키워드로 먼저 최적화할 수 있어 검색 결과의 품질을 유의미하게 향상시킵니다.

(2) 워크플로 노드 정의

상태 구조를 정의한 다음에는 워크플로를 구성할 다양한 노드들을 생성해야 합니다. LangGraph에서 각 노드는 특정 작업을 수행하는 Python 함수입니다. 이 함수들은 입력으로 현재 상태 객체를 받아 업데이트할 필드들을 담은 딕셔너리를 반환합니다.

노드들을 정의하기 전에 환경 변수 로드 및 대규모 언어 모델 인스턴스화를 포함한 프로젝트 초기화 설정을 완료합니다.

```python
import os
from dotenv import load_dotenv
from langchain_openai import ChatOpenAI
from langchain_core.messages import HumanMessage, AIMessage, SystemMessage
from tavily import TavilyClient

# .env 파일에서 환경 변수 로드
load_dotenv()

# 모델 초기화
# 이 llm 인스턴스는 모든 노드의 지능적 처리를 구동하는 데 사용됩니다
llm = ChatOpenAI(
    model=os.getenv("LLM_MODEL_ID", "gpt-4o-mini"),
    api_key=os.getenv("LLM_API_KEY"),
    base_url=os.getenv("LLM_BASE_URL", "https://api.openai.com/v1"),
    temperature=0.7
)
# Tavily 클라이언트 초기화
tavily_client = TavilyClient(api_key=os.getenv("TAVILY_API_KEY"))
```

이제 세 가지 핵심 노드를 하나씩 만들어 보겠습니다.

(1) 이해 및 쿼리 최적화 노드 (Understand & Query Node)

이 노드는 워크플로의 첫 번째 단계입니다. 사용자의 질문 의도를 이해하고 이를 최적화된 검색 키워드로 생성합니다.

```python
def understand_query_node(state: SearchState) -> dict:
    """1단계: 사용자 쿼리 이해 및 검색 키워드 생성"""
    user_message = state["messages"][-1].content

    understand_prompt = f"""사용자의 질문을 분석해 주세요: "{user_message}"
다음 두 가지 작업을 완료해야 합니다:
1. 사용자가 알고 싶어 하는 바를 간결하게 요약해 주세요.
2. 검색 엔진에 가장 적합한 키워드를 생성해 주세요. (한국어 또는 영어로 정확하고 정밀해야 합니다.)

형식:
이해: [사용자 요구사항 요약]
검색어: [최적의 검색 키워드]"""

    response = llm.invoke([SystemMessage(content=understand_prompt)])
    response_text = response.content

    # LLM의 출력을 파싱하여 검색 키워드 추출
    search_query = user_message # 기본적으로 기존 질문을 사용
    if "검색어:" in response_text or "Search terms:" in response_text:
        if "검색어:" in response_text:
            search_query = response_text.split("검색어:")[1].strip()
        else:
            search_query = response_text.split("Search terms:")[1].strip()

    return {
        "user_query": response_text,
        "search_query": search_query,
        "step": "understood",
        "messages": [AIMessage(content=f"다음 키워드로 검색을 수행합니다: {search_query}")]
    }
```

이 노드는 구조화된 프롬프트를 사용하여 LLM이 "의도 파악"과 "키워드 생성"이라는 두 가지 작업을 동시에 처리하도록 하며, 파싱된 전용 검색 키워드를 상태의 `search_query` field에 업데이트하여 다음 단계의 정밀 검색을 준비합니다.

(2) 검색 노드 (Search Node)

이 노드는 에이전트의 "도구 사용" 능력을 실행합니다. Tavily API를 호출하여 웹상에서 실시간 검색을 수행하며 기본적인 예외 처리를 포함합니다.

```python
def tavily_search_node(state: SearchState) -> dict:
    """2단계: Tavily API를 사용하여 실제 검색 실행"""
    search_query = state["search_query"]
    try:
        print(f"🔍 검색 중: {search_query}")
        response = tavily_client.search(
            query=search_query, search_depth="basic", max_results=5, include_answer=True
        )
        # ... (검색 결과 처리 및 포맷팅) ...
        # 여기서는 실제 코드에 구현된 결과를 축약해서 표현함
        search_results = str(response) # 포맷된 결과 문자열

        return {
            "search_results": search_results,
            "step": "searched",
            "messages": [AIMessage(content="✅ 검색 완료! 답변을 정리하는 중입니다...")]
        }
    except Exception as e:
        return {
            "search_results": f"검색 실패: {e}",
            "step": "search_failed",
            "messages": [AIMessage(content="❌ 검색 중 문제가 발생했습니다...")]
        }
```

이 노드는 `tavily_client.search`를 통해 외부 API 호출을 시작합니다. 예외 발생 시를 감안하여 `try...except` 블록으로 래핑되어 있으며, 검색 실패 시 `step` 상태를 `"search_failed"`로 업데이트하여 다음 노드가 폴백(fallback) 계획을 트리거할 수 있도록 유도합니다.

(3) 답변 생성 노드 (Answer Node)

마지막 답변 노드는 이전 검색의 성공 여부에 따라 다르게 대응하는 유연함을 발휘합니다.

```python
def generate_answer_node(state: SearchState) -> dict:
    """3단계: 검색 결과를 기반으로 최종 답변 생성"""
    if state["step"] == "search_failed":
        # 검색 실패 시 폴백 전략 실행 - LLM 자체 내부 지식으로 답변
        fallback_prompt = f"검색 API를 현재 사용할 수 없으므로, 내부 지식을 토대로 질문에 답해 주세요:\n사용자 질문: {state['user_query']}"
        response = llm.invoke([SystemMessage(content=fallback_prompt)])
    else:
        # 검색 성공 시 검색 결과를 종합하여 답변 생성
        answer_prompt = f"""다음 검색 결과를 바탕으로 사용자의 질문에 완전하고 정확하게 답변해 주세요:
사용자 질문: {state['user_query']}
검색 결과:\n{state['search_results']}
검색 결과를 종합하여 유용하고 증거에 기반한 답변을 작성해 주세요..."""
        response = llm.invoke([SystemMessage(content=answer_prompt)])

    return {
        "final_answer": response.content,
        "step": "completed",
        "messages": [AIMessage(content=response.content)]
    }
```

이 노드는 `state["step"]` 값을 검사하여 분기 로직을 처리합니다. 검색에 실패했다면 LLM의 자체 지식으로 답변을 구성하면서 오류 상황을 사용자에게 알립니다. 검색에 성공했다면 실시간 정보가 포함된 콘텍스트를 활용하여 시의성 있고 확실한 답변을 완성합니다.

(4) 그래프 구축

모든 노드들을 연결합니다.

```python
from langgraph.graph import StateGraph, START, END
from langgraph.checkpoint.memory import InMemorySaver

def create_search_assistant():
    workflow = StateGraph(SearchState)

    # 노드 추가
    workflow.add_node("understand", understand_query_node)
    workflow.add_node("search", tavily_search_node)
    workflow.add_node("answer", generate_answer_node)

    # 선형 프로세스 설정
    workflow.add_edge(START, "understand")
    workflow.add_edge("understand", "search")
    workflow.add_edge("search", "answer")
    workflow.add_edge("answer", END)

    # 그래프 컴파일
    memory = InMemorySaver()
    app = workflow.compile(checkpointer=memory)
    return app
```

이 스크립트를 실행한 후, 실시간 정보가 필요한 질문(예: 1장의 실습 질문인 `내일 베이징에 가는데 날씨는 어떤가요? 가볼 만한 추천 관광지가 있나요?`)을 입력하면 터미널을 통해 에이전트의 구체적인 "사고" 프로세스가 출력되는 것을 볼 수 있습니다:

### 6.5.3 LangGraph의 장단점 분석

모든 프레임워크는 각자의 도메인과 트레이드오프를 가집니다. 이 섹션에서는 LangGraph의 핵심적인 강점과 실제 현업 애플리케이션 구축 시 직면하는 한계를 객관적으로 비교합니다.

(1) 장점

- 지능형 검색 어시스턴트 사례에서 본 것처럼, LangGraph는 전체 실시간 Q&A 흐름을 상태, 노드, 에지로 이루어진 "순서도(flowchart)"로 구체적으로 정의합니다. 이 설계의 가장 큰 매력은 **높은 제어성과 예측 가능성**입니다. 개발자는 에이전트의 모든 조치 단계를 세밀하게 제어할 수 있어 신뢰성과 감사(audit) 기능이 필수적인 프로덕션 수준 업무에 적용하기 매우 유리합니다. 가장 강력한 기능은 **순환 구조(cycles)의 네이티브 지원**입니다. 조건부 에지를 통하여 "성찰-수정(reflection-correction)" 루프를 손쉽게 구축할 수 있습니다. 예를 들어, 검색이 실패했을 때 백업 계획으로 우회하는 경로는 에이전트의 자체 최적화와 결함 극복에 핵심적인 장치가 됩니다.
- 또한 각 노드가 독립적인 Python 함수이므로 **높은 모듈성**을 가지며, 중간에 인간 검토(human review)를 대기하는 노드를 삽입하기도 매우 명확하여 현실적인 "루프 속의 인간" 협업을 구현하는 탄탄한 토대를 제공합니다.

(2) 한계

- 대화 기반 프레임워크와 비교할 때, LangGraph는 개발자에게 상대적으로 많은 **보일러플레이트 코드(boilerplate code)**를 요구합니다. 상태, 노드, 에지를 정의하고 빌드하는 일련의 작업들은 단순한 작업의 경우 개발 과정을 무겁고 성가시게 만들 수 있습니다. 개발자는 무엇을 할지(what to do)보다 과정을 어떻게 제어할지(how to control)에 더 머리를 싸매야 합니다. 미리 약속된 워크플로를 따르기 때문에 행동을 완벽하게 통제할 수 있는 대신, 대화형 에이전트 특유의 역동적이고 **"창발적인" 상호 작용(emergent interaction)**은 다소 희석됩니다. 이 프레임워크의 존재 이유는 불확실한 사회적 협업 시뮬레이션이 아니라 확정적이고 믿을 수 있는 업무 프로세스의 철저한 수행에 있습니다.
- 디버깅 프로세스 또한 만만치 않습니다. 비록 대화 이력을 일일이 추적하는 것보다 흐름은 더 선명하지만, 노드 내부의 로직 에러, 노드 간에 전달되는 상태 데이터의 변질, 혹은 에지 분기 조건 판단의 오류 등 여러 지점에서 장애가 일어날 수 있어 개발자에게 그래프의 전반적인 작동 방식에 대한 깊은 종합적 이해를 요구합니다.

## 6.6 본장 요약

이 장에서 우리는 구체적인 사례를 통한 실습을 통해 에이전트 프레임워크의 가장 전위적인 형태들을 경험했습니다.

각 프레임워크가 에이전트 구축 문제를 저마다 고유한 방식으로 다루고 있음을 보았습니다:

- **AutoGen**은 복잡한 협업을 여러 역할 간의 자동화된 "그룹 채팅"으로 추상화하며, 본질은 "대화를 통한 협력 유도"에 있습니다.
- **AgentScope**는 산업 등급 애플리케이션의 강건성과 확장성에 초점을 두어, 대규모 동시성과 분산 처리 기능을 갖춘 멀티 에이전트 시스템을 위한 튼튼한 엔지니어링 토대를 제공합니다.
- **CAMEL**은 매우 정교한 "역할 수행(Role-Playing)"과 "인셉션 프롬프팅" 패러다임을 통해 최소한의 코드로 두 에이전트의 깊고 자율적인 협업을 이끌어 낼 수 있음을 증명합니다.
- **LangGraph**는 좀 더 근본적인 "상태 머신" 개념으로 복귀하여, 명시적인 그래프 구조를 기반으로 개발자에게 프로세스 제어권을 엄격하게 양도하고 특히 순환 기능을 지원해 반성하고 교정 가능한 에이전트 구축의 문을 엽니다.

이 프레임워크들을 심도 있게 비교 분석하면서 우리는 하나의 근본적인 절충 지점을 포착할 수 있습니다: **"창발적 협업(emergent collaboration)"과 "명시적 제어(explicit control)" 사이의 선택**입니다. AutoGen과 CAMEL은 에이전트의 "역할"과 "목표" 정의에 치중하고 단순한 대화 규칙에서 다채로운 협업이 자연스레 피어나도록 방임합니다. 이는 인간의 소통 양식에 더 가까우나 예측과 디버깅이 어렵습니다. 반면 LangGraph는 모든 단계와 경로 전이 조건을 개발자가 직접 조율하도록 요구하여, 창발적이고 예외적인 즐거움은 희생하는 대신 높은 신뢰성, 예측 가능성, 그리고 관찰 가능성을 손에 넣습니다.
여기에 AgentScope는 매우 중요한 두 번째 축인 **엔지니어링(Engineering)**을 가리킵니다. 우리가 어떠한 협업 모델을 채택하든, 실험실의 장난감을 넘어 실제 비즈니스 가치로 이행하기 위해서는 동시성 제어, 장애 극복, 분산 서버 구조 등 혹독한 인프라 도전을 극복해야만 합니다. AgentScope는 이러한 문제를 원천적으로 해결하고자 태어났으며, "돌아간다"에서 "안정적으로 서비스한다"로 나아가는 중대한 엔지니어링 도약을 상징합니다.

에이전트를 짓는 방법은 단 하나가 아닙니다. 이 장에서 깊이 다룬 프레임워크들의 아키텍처 철학을 내재화한다면, 단순한 "도구 사용자"를 넘어 프레임워크 설계의 트레이드오프를 자유자재로 다루는 아키텍트로 거듭날 것입니다.

다음 장에서 우리는 모든 이론과 실무를 종합하여, 처음부터 끝까지 나만의 에이전트 프레임워크를 직접 설계하고 코딩하는 본 튜토리얼의 백미로 나아갈 것입니다.

   - `LangGraph`는 본질적으로 순환 구조를 쉽게 짜기 위해 태어났습니다. 이 강점을 극대화할 수 있는 응용 업무 시나리오(예: "코드 자동 생성-테스트 빌드-디버깅 및 수정" 루프, 혹은 "학술 논문 집필-피어 리뷰-피드백 수정" 루프 등)를 설계하고, 전체 그래프 구조와 주요 노드의 역할을 기술해 보세요.

6. 프레임워크 선택은 에이전트 제품 기획 시 가장 치명적인 의사 결정 중 하나입니다. 당신이 `AI` 테크 기업의 리드 아키텍처라고 가정할 때, 회사에서 아래 3가지 유형의 에이전트 서비스를 상용화하려 합니다. 각 비즈니스 유형에 가장 이상적인 프레임워크를 선택하고(`AutoGen`, `AgentScope`, `CAMEL`, `LangGraph`, 혹은 프레임워크 없이 백그라운드부터 직접 구현) 그 아키텍처적 근거를 설명해 주세요:

   **서비스 A**: 지능형 대화 고객 센터 시스템. 초당 1,000건 이상의 대규모 동시 요청(high concurrent requests)을 매끄럽게 처리해야 하며, 지연 시간(latency)은 2초 미만으로 묶어야 합니다. 시스템은 365일 24시간 중단 없이 지속 가동되어야 하며 수평적 확장성(horizontal scaling)을 필수로 요구합니다.

   **서비스 B**: 학술 연구 및 논문 집필 보조 에이전트. "문헌 리서치 담당 에이전트"와 "학술 논문 집필 담당 에이전트"가 유기적이고 밀접하게 협동해야 하며 선행 연구 조사, 실험 설계 조율, 데이터 해석, 드래프트 집필을 스스로 완수해야 합니다. 에이전트들은 여러 차례 상호 브레인스토밍 토론을 자율적으로 전개해 나가며 업무를 해결해야 합니다.

   **서비스 C**: 금융 여신 신용 대출 심사 시스템. 정해진 프로세스에 맞춰 대출 심사를 한 단계씩 넘어가야 합니다: 서류 적격 여부 판정 → 신용 위험 종합 검토 → 승인 한도 연산 → 금융 규제 컴플라이언스 체크 → 관리자 최종 승인 확인 → 대출 실행 결정. 각 단계는 흑백이 뚜렷한 감사 조건과 정밀한 브랜치 로직을 통과해야 하며, 전 과정은 완벽하게 추적(trace)되고 감사(audit)될 수 있어야 합니다.
